# CLAUDE.md — digest-bot-latvia-ru

Этот документ — единая точка правды по проекту для агентов (Claude Code и других),
а также шпаргалка для самого владельца. Раздел «Текущие фичи» обновляется

## 1. Что это за проект

Telegram-бот, собирающий новости, которые владелец форварит в течение дня в
канал-источник, и по команде `/digest` генерирующий итоговый дайджест с
пересказами через Claude API, публикуя его в целевой канал.

Работает в паре с отдельным проектом `allnews-latvia-ru` (полностью независимый,
живёт рядом на том же VPS, агент в него **не лезет** ни при каких обстоятельствах).

Владелец и единственный пользователь команд — Kim (OWNER_ID).

## 4. Инварианты данных

### digest_state.json

Формат не реструктурируем без явной задачи. Поля каждой записи:

```json
{
  "id": 1,
  "url": "https://...",
  "title": "Заголовок (если есть)",
  "source": "Источник (если удалось определить)",
  "tag": "выборы" | null,
  "level": 0,
  "added_at": "2026-06-28T14:30:00"
}
```

- **`id`** — стабильный, монотонно растущий. После `/remove` остаются «дыры»
  (id не пересчитываются). После `/maketop` порядок в списке меняется, но **id
  остаются прежними**. Это сознательное решение, удобно для серии команд:
  можно делать `/maketop` подряд, не пересматривая `/list`.
- **`tag`** — строка без `#` и без `+`. `+` отдельно живут в `level`.
- **`level`** — уровень развёрнутости пересказа для тег-группы:
  `0` (compact, 1-2 предложения), `1` (medium, 3-5), `2` (extended, 1-2 абзаца),
  `3` (large, несколько абзацев). Определяется числом `+` после хэштега:
  `#тема` = 0, `#тема+` = 1, `#тема++` = 2, `#тема+++` = 3.
  В группе берётся **максимальный** уровень из всех её участников.

При смене даты файл переинициализируется (новый день — новый список).

### Тег-группы

- `MIN_GROUP_SIZE = 2` (`config.py`).
- Если новостей с одним тегом ≥ MIN_GROUP_SIZE → выводятся сводным пересказом
  (один блок «📌 …, Источники: …»).
- Если < MIN_GROUP_SIZE → каждая новость становится одиночным блоком.
- Порядок групп и одиночек в дайджесте — **по позиции первого элемента в
  списке `items`** (см. логику `build_digest`). Это критичный инвариант для
  работы `/maketop`.

## 5. Команды бота

Все команды — **только для OWNER_ID**, остальные пользователи игнорируются молча.

- `/start` — приветствие и help-текст.
- `/list` — пронумерованный список накопленных за день новостей.
- `/tag N hashtag[+...]` — пометить новость #N тегом (с уровнем по числу `+`).
- `/tag N` — снять тег с новости #N.
- `/remove N` — удалить новость #N из списка.
- `/maketop N` — поднять новость #N наверх списка. Если у новости есть тег,
  вместе с ней наверх едет вся её тег-группа: выбранная становится первой
  в группе, остальные следуют в исходном относительном порядке.
- `/clear` — полная очистка списка.
- `/digest` — собрать и опубликовать дайджест из текущего списка.

### Канал-источник vs целевой канал

- **Канал-источник** (где владелец собирает форварды): `@fghjlllkjd`.
  Бот слушает его и парсит каждый пост.
- **Целевой канал** (куда публикуется готовый дайджест): `@kim_test_channel`.
  Исторически назывался «тестовым», сейчас это рабочая промежуточная
  площадка — владелец вручную форвардит готовый дайджест в публичные каналы.

### Формат постов в канале-источнике

`🕐 HH:MM | Источник 📰 Заголовок 🔗 [🔁] + ссылка на следующей строке`.
Парсер также поддерживает старый формат без эмодзи. Блок «Ранее по теме»
игнорируется.

### Пометка хэштегом

Хэштег приходит **отдельным сообщением до форварда**. Бот запоминает его и
привязывает к следующему форварду. Это паттерн «форвард с комментарием»,
используется потому что в Telegram нельзя добавить текст к форварду напрямую.

## 6. Источники и обходы блокировок

### Текущие источники (все рабочие)

TVNET, Jauns, BB.lv, Delfi, LSM.

### LSM (rus.lsm.lv)

Блокирует датацентровый IP Hetzner (403). Решение — SOCKS5-прокси через
Tailscale exit node на телефоне владельца (Redmi Note 10 5G).

- На VPS работает второй userspace-экземпляр Tailscale как systemd-юнит
  `tailscale-proxy`, поднимающий SOCKS5 на `localhost:1055`.
- Системная маршрутизация VPS **не затронута** (это критично — иначе сломается
  параллельный newsbot).
- `config.py`: `PROXY_URL = "socks5h://localhost:1055"`, `PROXY_DOMAINS = ["lsm.lv"]`,
  `PROXY_FETCH_TIMEOUT = 20`.
- `article_fetcher.py` направляет через прокси только домены из `PROXY_DOMAINS`,
  остальное — напрямую.
- Если телефон офлайн или Tailscale exit node выключен — LSM просто не грузится,
  попадает в «не вошло». Это **ожидаемое поведение**, ловить не нужно.

### Delfi (rus.delfi.lv)

Не блокирует, но использует Nuxt.js (SPA). Trafilatura вытаскивает JS-мусор
вместо текста.
- `config.py`: `BS4_ONLY_DOMAINS = ["delfi.lv"]`.
- Для этих доменов `article_fetcher.py` пропускает trafilatura и сразу идёт
  в BeautifulSoup (селектор `article p`).

### Логика загрузки статей (`article_fetcher.py`)

1. Скачивание через `requests` с браузерными заголовками. **Brotli в
   `Accept-Encoding` НЕ указываем** — иначе получаем нечитаемый мусор.
2. Для доменов из `BS4_ONLY_DOMAINS` — сразу BS4, минуя trafilatura.
3. Для остальных — сначала trafilatura, потом BS4 как fallback.
4. BS4 идёт по списку селекторов (`article.article-data p` для jauns, `article p`
   и т.д.), берёт абзацы ≥40 символов.
5. Для доменов из `PROXY_DOMAINS` запрос через SOCKS5.
6. Текст обрезается до `ARTICLE_MAX_CHARS` (8000).

## 10. Текущие фичи

### Фича 3 (reliability): jauns периодически не загружается

**Симптом:** примерно 1-2 раза в неделю статья с `jauns.lv` попадает в блок
«⚠️ Не вошли в дайджест (не удалось загрузить)». В браузере статья тоже
открывается с заметной задержкой. Точная причина пока неизвестна.

**Стратегия:** мягкая правка без агрессивных изменений. Этап 1 — собрать
данные о фейлах. Этап 2 (если данных будет недостаточно после нескольких
недель) — точечная починка.

**Файлы:** `article_fetcher.py`, `config.py`.

**Что делаем:**

1. **Расширенное логирование пути загрузки для `jauns.lv`** (на уровне INFO,
   видно в `bot.log`):
   - URL и домен.
   - Время начала запроса и общая длительность загрузки (`elapsed_ms`).
   - HTTP status code и размер тела ответа (байт).
   - Какой путь экстракции сработал: `trafilatura` или `bs4_fallback`.
   - Длина итогового текста (символов) после обрезки до `ARTICLE_MAX_CHARS`.
   - Если итог пустой — какие селекторы BS4 пробовались и сколько абзацев
     каждый вернул (отдельной DEBUG-строкой, чтобы можно было поднять
     уровень для одной диагностики).

2. **Отдельный таймаут для `jauns.lv`:**
   - Добавить в `config.py` константу `JAUNS_FETCH_TIMEOUT = 30` (или
     ввести более общий механизм per-domain таймаута — на усмотрение
     агента-генератора, но в `CLAUDE.md` потом фиксируется выбранное
     решение).
   - Базовый таймаут для остальных доменов не меняется.

3. **Один retry с паузой при таймауте или 5xx для `jauns.lv`:**
   - Если первый запрос упал по `requests.Timeout` или вернул 5xx —
     сделать ОДНУ повторную попытку через 3 секунды.
   - Retry применяется **только** к jauns. Для остальных доменов поведение
     не меняется.

**Что НЕ делаем (намеренно):**
- Не переносим jauns в `BS4_ONLY_DOMAINS` (нет данных, что trafilatura лажает
  на их контенте — только что медленно отдают).
- Не добавляем retry для всех доменов глобально.
- Не меняем браузерные заголовки.
- Не вводим прокси для jauns.

**Что НЕ трогать:**
- Логику LSM (прокси) и Delfi (BS4_ONLY).
- Сигнатуру публичных функций `article_fetcher.py`, которые вызываются из
  `digest_builder.py`/`bot.py` (без необходимости).

**Критерии готовности:**
- В логах при загрузке jauns видны все перечисленные поля.
- Виден явный таймаут 30 секунд при загрузке jauns (если уложился быстро —
  таймаут не сработает, и это нормально).
- Виден retry при искусственно вызванном таймауте (можно проверить
  локально через `diagnose_url.py` с временной правкой URL на несуществующий
  поддомен jauns).
- Регрессий по другим источникам нет: TVNET/BB.lv/Delfi/LSM грузятся как
  раньше.

### Фича 4 (UX): ошибки загрузки — только в DM владельцу

**Что меняется:** блок «⚠️ Не вошли в дайджест (не удалось загрузить)»
полностью **исчезает из публикуемого поста** в целевом канале. Если в
момент сборки дайджеста были невошедшие новости, бот отправляет
**отдельное сообщение в личку OWNER_ID** со списком этих новостей и причинами.

**Файлы:** `digest_builder.py`, `bot.py` (хендлер `cmd_digest`).

**Текущее поведение:**
- Блок «Не вошли» подклеивается в конец публикуемого дайджеста.
- Дополнительно, если **сама публикация в канал упала** — бот шлёт в личку
  владельцу `.txt`-файл с текстом дайджеста как страховку.

**Новое поведение:**
- Блок «Не вошли» **не добавляется** в текст для публикации в канал.
- Если `failed_items` непустой после сборки — после успешной публикации
  основного дайджеста бот шлёт в личку OWNER_ID отдельное сообщение в
  формате:
  ```
  ⚠️ Не вошли в дайджест за DD.MM.YYYY:
  • <Заголовок или URL> — <причина>
  • ...
  ```
- Если `failed_items` пустой — ничего в личку не шлётся (никаких «всё ОК»
  сообщений).
- **Страховочный `.txt`-файл при ошибке публикации — остаётся как был.**
  Это про другой сценарий (упала публикация целиком), его трогать не надо.

**Edge cases:**
- Если упала и публикация, и были невошедшие — сначала отрабатывает
  страховочная отправка `.txt`, отдельное сообщение про невошедшие шлётся
  после неё (тоже в личку).
- Если в личку отправить не удалось — пишем ошибку в лог, не падаем.

**Что НЕ трогать:**
- Сам формат дайджеста (шапка, блоки, разделители).
- Логику отправки страховочного `.txt` при падении публикации.
- Сохранение дайджеста в файл `digest_YYYY-MM-DD_HH-MM-SS.txt` перед
  публикацией.

**Критерии готовности:**
- В публикуемом посте никогда нет блока «Не вошли».
- Если были невошедшие — пришло отдельное сообщение в личку OWNER_ID со
  списком и причинами.
- Если невошедших не было — ни одного дополнительного сообщения в личку.
- При падении публикации `.txt`-файл шлётся как раньше.

## 11. Сделано

*(Раздел пополняется одной строкой на фичу по мере закрытия.)*

- `/maketop N` — поднятие новости (и её тег-группы) наверх списка.
- Косметика: `⚠️⚠️⚠️` в шапке дайджеста, `▪️`-сепаратор между источниками
  в групповых блоках.
- Перенос разработки с Windows на Linux Mint (Ideapad 13, VS Code + Claude Code).
- Косметика: одна пустая строка между блоками дайджеста (вместо двух).
- Кликабельные заголовки в `/list` (Markdown-ссылки на оригинальные URL),
  без обрезки длины. Пустой title → URL как plain text.
- Защита публикации от транзиентных таймаутов: write/read timeout 30 сек +
  retry с экспоненциальным backoff (2, 4, 8 сек) на каждую часть отдельно.
  Сохранена идемпотентность — уже опубликованные части не пересылаются.
